HL7 Messages


HL7 messages transmit data between healthcare information systems. There are two main standards currently in use: HL7v2 and HL7 FHIR.

FHIR (Fast Health Interoperable Resources) is an HL7 standard currently under development to allow for more Agile and RESTful approach to healthcare IT interoperability.

HL7v2 is the standard used by most extant healthcare systems used today, though many are making efforts to transition to HL7 FHIR.

We'll take a brief look at HL7v2. If you're interested in learning more about HL7 FHIR, please refer to the document section below.

An HL7v2 message consists of a group of segments in a defined sequence.

Here's an example HL7v2 message:

MSH|^~\&|SENDING_APP|SENDING_FACILITY|RCVNG_APP|RCVNG_FACILITY|20110613083617||ADT^A01|934576120110613083617|P|2.3||||
EVN|A01|20110613083617|||
PID|1||135769||DENT^ARTHUR^||19281118|M|||123 Hitchhiker Way^^Tallahassee^FL^32308||(407)939-1289^^^dontpanic@space.com|||||1719|99999999||||||||||||||||||||
PV1|1|O|||||^^^^^^^^|^^^^^^^^

The first field in the first segment defines the segment type. In this message it's MSH, so this segment is the message header.

The second field in the first segment defines the separator characters. These are used to divide the message fields into components and sub-components, define repeated components, and define the escape character.

Here are the standard separator characters used in most HL7 messages:

Separator Character Function
| Field separator, aka pipe
^ Component separator, aka hat
& Sub-component separator
~ Field repeat separator
\ Escape character

The segments or groups of segments can be optional, required, and/or repeatable (i.e., the cardinality). The type of characters surrounding the segments determine the cardinality.

Here's how cardinality is represented in an HL7 message:

Cardinality Surrounding Brackets
Required No brackets
Optional [ ]
Repeating { }
Optional and Repeating [{ }]

Every HL7 message has a type that defines the purpose of the message. Message types are identified by a three-character code, and are used in conjunction with a trigger event.

An HL7 trigger event is a real-world event that initiates communication and the sending of a message, and is shown as part of the message type. Both the message type and trigger event are found in the MSH-9 field of the message.

For example, as in the message above, the MSH-9 field might contain the value ADT-A01. This means that ADT is the HL7 message type, and A01 is the trigger event. In the HL7 Standard, an ADT-A01 message is known as a "patient admit" message.

There is much more to be said about the format of HL7 messages. We haven't even discussed the evil Z segments or HL7 FHIR.
